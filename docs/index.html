1 Introduction

JNS is a dynamically typed transpiled Programming Language designed for building maintainable software. It's based off Python and its design has also been influenced by V, Rust, C and (Believe it or not) ALGOL.

JNS is not a very hard language. Going through this documentation will take you about an hour, and by the end of it you will have pretty much learned the entire language. The language promotes writing simple and clear code with minimal abstraction. Despite being simple, JNS gives the developer a lot of power. Anything you can do in other Python, you can do in JNS.

1.1 Installation from Source

The best way to get the latest and greatest JNS, is to install it from source. It is easy, and it takes only about a minute:

git clone https://github.com/JaneScriptLang/PyJNS

cd PyJNS

./prepare -–with-jmake && make # Assuming you want JMake installed, on Linux

./build/python ./bootstrap-win.py

1.2 Testing and running samples

Make sure JNS can run itself, by executing the following commands in the cmd.exe Shell (Inside the JNS Install Directory):

$ jns ./bin/test.jns


$ cd examples

$ jns hello_world.jns

$ jns wc/word_counter.jns wc/beemovie.txt

$ jns se/se.jns

2.2 Hello, World!

As in every other language, your very first Program will be a „Hello, World!“ Script… literally, just a single line, outputting „Hello World“ to your screen. In JNS, this statement is exactly:

print(„Hello, World!“);



2.3 Taking input and using Variables

Variables are a way to store data. Since JNS is dynamically typed, you won’t have to add the Type of the Variable to the declaration. What in C looks like int age = 0; in JNS looks like age = 0; . Look at how simple this code is:

#include <stdlib>

name = input(„What’s your Name?“);

printf(„Hello, %!“, (name,));


2.4 Comments

A Comment („//“) will remove all text after the comment decorator. So this: print(„Hello, World!“); // Will print „Hello, World!“ works the same as just print(„Hello, World!“); Comments are useful when writing complex code, so you don’t lose overview of what you’re writing.

3.2 JNS Control Statements

JNS has three categories of control Statements: Conditional Statements, Loop Statements and Pragma-Statements

  • Conditional Statements, which run different statements for different data values.

The Conditional Statements are IF, ELSE IF and ELSE

  • Loop Statements, which repeat Segments of code n times, whereas n can be a definite number or a call to a Function that returns an integer. The exception to this are WHILE-Loops, that repeat a Code segment until a condition is met.

    The Loop Statements are FOR/UNTIL, FOREACH and WHILE

  • Pragmas, which are parameters passed into the Transpiler after reading the script, but before transpiling anything to Pythonic Code. Pragmas consist of a Name and a Value, providing extra required details to the C compiler. These extra details can be anything that was somehow not passed within the program or the code logic. A Pragma Declaration looks like this: #pragma name,value . A List of PyJNS-Specific Pragmas can be found in 3.6

Conditional Statements

The Conditional Statements, IF and ELSE run different statements for different Data Values



The If/Then Statement

The If/Then Statement either runs or skips a Sequence of one or more Statements, depending on a condition. It follows the following structure:

if(condition){

print(„Statements here...“);

}

If condition is equal to a boolean True, the statements run; otherwise the if/then-Statement does nothing.



TIP: Avoid redundant IF/THEN/ELSE Statements such as:

if(new_balance < minimum_balance){

overdrawn = True;

} else {

overdrawn = False;

}

Instead, assign the Boolean Value of condition directly to a Boolean Variable:

overdrawn = (new_balance < minimum_balance);

A Boolean Variable is either True or False. None has the same properties as False, so avoid statements like this:

if(overdrawn==True){

throw(InsufficientFundsError());

}

Instead, write:

if(overdrawn){

throw(InsufficientFundsError());

}


Ex. 3.3.1: IF/THEN

The following function will return a success Response if and only if the user is authenticated

fn dashboard(req, res){

session = req.get_session();

if(session[„authenticated“]){

res.write(„Authenticated“);

res.code(200);

}

}


The If/Then/Else Statement

The If/Then/Else Statement follows the following structure:

if(condition){

print(„If-Statements here...“);

} else {

print(„Else-Statements here...“)

}

If condition is equal to a boolean True, the if-statements run; otherwise the else-Block will run. If/Then and If/Then/Else Statements can be nested.



Ex. 3.3.2: IF/THEN/ELSE

The following function will return a success Response if the user is authenticated, else redirect him to the login Page



fn dashboard(req, res){

session = req.get_session();

if(session[„authenticated“]){

res.write(„Authenticated“);

res.code(200);

} else {

res.location(„/login“);

}

}



Loop Statements

The Loop Statements, F OR and WHILE repeat segments of code several times, until an exit condition is met.



The FOR/UNTIL Loop

Unlike Python, the JNS-For-Loop is a Counting loop, meaning that it doesn’t iterate through Iterables, it counts up (or down). It follows the following Structure:



for counter=0 until 10 step 1{

print(„Statements here...“);

}

This Loop Starts at 0 and repeats the statements until counter + 1 = 10



Example 3.3.3: FOR/UNTIL

The following Code is a practical example of calculating the Factorial of n

fn fac(n){

total = n;

for i=1 until n step 1{

total *= i;

}

return total;

}

The FOREACH Loop

Most Programmers will be used to some kind of For/Each Loop in the Modern day, so they don’t have to write countless and countless For/Until Loops. In JNS, For/Each-in works in 2 Ways: The recommended one and the cheaty one.

The Recommended one:



const myArray = [1,2,3,4,5];

// Method 1 – Using the Arrays Library

#include <arrays>

foreach(myArray, lambda i: print(i));

//Method 2

array.items.foreach(lambda i: print(i));

JNS reads For/Until loops like mentioned above. However, to avoid getting stuck, it adds a marker to each transpiled For-Loop. If the word „#transpiled“ is found, the Transpiler will assume that the Loop already has been processed. That’s why you can write #transpiled at the end of a Python-For Loop

The WHILE/DO Loop

The WHILE-Loop repeats Sequences of Code until a condition is met. „Condition being met“ means it has a Boolean Value of TRUE. This can be something like input == „Hello“ or any other condition (See IF/THEN-Statements for reference). Generally, WHILE-Loops look like this:

while(condition){

print(„Statements...“);

}

3.3 JNS Subprograms

A Subprogram is a named JNS Block that can be invoked repeatedly.

A Subprogram is either a Procedure or a function. Typically, you use procedures to perform an action and functions to compute and return a value.

Reasons to use Subprograms

Subprograms support the development and maintenance of reliable, reusable code

with the following features:

  • Modularity

Subprograms let you break a program into manageable, well-defined modules.

  • Easier Application Design

When designing an application, you can defer the implementation details of the subprograms until you have tested the main program, and then refine them one step at a time. (To define a subprogram without implementation details, use the ... Operator, as in Example 3.4.1.)

  • Maintainability

You can change the implementation details of a subprogram without changing its invokers.

  • Packageability

Subprograms can be grouped into packages, whose advantages are explained in "Reasons to Use Packages".

  • Reusability

Any number of applications, in many different environments, can use the same

package subprogram or standalone subprogram.

Declaration and Invocation



You can create a subprogram either inside a JNS block (which can be another

subprogram), or at Global Level.

A subprogram created inside a JNS block is a nested subprogram. You can either

declare and define it at the same time, or you can declare it first and then define it later in the same block (see "Forward Declaration").

A subprogram created inside a package is a package subprogram. You declare it in

the package specification and define it in the package body. It is stored in the database until you forget() the package. (Packages are described in JNS Packages and R4-Notation.)



A subprogram created at global level is a standalone subprogram. You create it with

the fn-statement.

fn main() {

print(„Hello World“);

}

You can also specify parameters that will be passed into the subprogram’s body:

fn add(a,b){

return a+b;

}

A subprogram invocation has this form:

subprogram_name ( parameter, parameter... )

If the subprogram has no parameters, or specifies a default value for every parameter, you can either omit the parameter list or specify an empty parameter list.

A procedure invocation is a JNS statement. For example:

raise_salary(employee_id, amount);

A function invocation is an expression. For example:

new_salary = get_salary(employee_id);

if (salary_ok(new_salary, new_title)){

...;

}


Recursive Subprograms

Recursive Subprograms are Subprograms that call themselves with similar, yet different Parameters. A Recursive subprogram usually first checks if a condition is met („Is the Problem simple?“). If so, it stops and returns the problem immediately, else it simplifies the Problem and calls itself with the simplified Problem. Recursions can look like this:

fn fac(n):

if(n<=1){

return n;

}

else {

return fac(n-1)*n;

}

}

or like this:

@recursive(lambda n: n > 1)

fn fac(n){

return fac(n-1)*n;

}

Subprograms in Packages

You can use the #include Statement to import a package. This will lead to all Subprograms inside this Package to be added to your global Variables. If you're using the latest intern Build, you can also use R4-Syntax. Thanks to the R4-Notation, you can now also write this code to include specific Subprograms from a package:

import fn1, fn2 from module;



3.4 JNS Packages and R4-Notation

A package is a namespace object that groups logically related JNS structures, variables, constants, subprograms and exceptions. A package is compiled and stored in the database, where many applications can share its contents.

A package always has a specification, which declares the public items that can be

referenced from outside the package. If the public items include cursors or subprograms, then the package must also have a body. The body must define queries for public cursors and code for public

subprograms. The body can also declare and define private items that cannot be

referenced from outside the package, but are necessary for the internal workings of the package. Finally, the body can have an initialization part, whose statements initialize variables and do other one-time setup steps, and an exception-handling part. You can change the body without changing the specification or the references to the public items; therefore, you can think of the package body as a black box.

Reasons to use Packages

Packages support the development and maintenance of reliable, reusable code with

the following features:

  • Modularity

Packages let you encapsulate logically related types, variables, constants,

subprograms, cursors, and exceptions in named PL/SQL modules. You can make each package easy to understand, and make the interfaces between packages simple, clear, and well defined. This practice aids application development.

  • Easier Application Design

When designing an application, all you need initially is the interface information in the package specifications. You can code and compile specifications without their bodies. Next, you can compile standalone subprograms that reference the packages. You need not fully define the package bodies until you are ready to complete the application.

Declaration (Pre-R4)

The R4-Notation (Revision 4) adds lots of new functionality to JNS, but at this point it’s still in early beta. If you’re using the latest stable Version of JNS, you can create a package using the distlib Library. For this, you need a file called __package__.jns . This File needs to follow this pattern:

#include <distlib>

package = info();

package.name = „Package Name“;

package.include_name = „include_name“;

package.author = „your name“;

// You can also write package = getuname();, which will just

//use your login username.

package.includes = [„file1“, „file2“]; //write all the include-Names you want to use in your Package

make_package(package);



The above code will create a long, long, long File that’s named „[include_name].jns“. This File contains all the code you specified in package.includes. You can now #include <include_name> to invoke your Package.

Declaration (Post-R4)

Revision 4 is a currently experimental Release of JNS. It simplifies many things, for example the package declaration. To create a Package now, create a file that is named include_name.jns, with the following code:

package author:name;

namespace file1;

namespace file2;

The first line declares your Package Metadata. Then you write namespace module ; For each file you want to include. Note that R4-Packages can only be included using the R4-Notation: import file1, file2 from package;


3.5 JNS Pragma List

Pragmas are used to pass information to the Transpiler at runtime. This list contains all Pragmas that have an effect on the TP:

  • NP_INCLUDE, <module name>: If <module_name> is imported at any point inside the code, it will get stripped out. Useful when facing compatibility Issues.

  • PO_PLATFORM, [win32|linux|darwin]: Assure that the Platform the code is being executed on is win32 (Windows), linux (Linux) or darwin (MacOS)

  • NP_PLATFORM, [win32|linux|darwin]: Assure that the Platform the code is being executed on is not win32 (Windows), linux (Linux) or darwin (MacOS)

  • RD_STDOUT, <filename>: Redirect STDOUT to a File

  • RD_STDERR, <filename>: Redirect STDERR to a File
